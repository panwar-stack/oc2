# Publish OC2 To npm

OC2 releases are published manually through the `publish` GitHub Actions
workflow. The workflow builds and validates all platform packages, publishes
the native packages, and publishes the `oc2-ai` wrapper last. Windows
executables are intentionally published unsigned.

## Prerequisites

Before preparing a release:

- have permission to create tags and dispatch workflows in `panwar-stack/oc2`;
- confirm the `oc2-ai` package and every `oc2-<platform>` package produced by
  `packages/opencode/script/build.ts` are controlled by the intended npm
  organization or maintainers; and
- confirm npm trusted publishing is configured as described below.

Use a canonical semver such as `1.2.3` and a lowercase npm dist-tag such as
`latest` or `next`. The workflow rejects a version with a leading `v` and
rejects unsafe or semver-like dist-tags.

## Bootstrap npm And Trusted Publishing

npm trusted publishing is configured per package. Before the first workflow
release, an npm owner must reserve or bootstrap all 13 package names: the
`oc2-ai` wrapper and the 12 native package names generated by the build. If npm
requires an initial authenticated publication before trusted publishing can be
enabled, perform that bootstrap through npm's current supported process and
verify package ownership before creating a release tag.

For every package, add a GitHub Actions trusted publisher in npm with:

- organization or user: `panwar-stack`;
- repository: `oc2`;
- workflow filename: `publish.yml`; and
- environment: unset, unless the workflow is also updated to use that GitHub
  environment.

Repeat this configuration for the wrapper and every native package. npm's
bootstrap and configuration screens may change independently of this
repository; use the current trusted-publishing documentation if they differ
from the fields above.

The publish job requests `id-token: write`, uses npm trusted publishing, and
enables provenance. Do not add an `NPM_TOKEN`: the workflow is intentionally
authenticated to npm through GitHub OIDC.

## Tag And Dispatch A Release

Create an immutable tag on the exact commit to publish, push it, then dispatch
the workflow from that tag. The input version must exactly match the tag name
without its `v` prefix.

```sh
git tag v1.2.3 <release-commit>
git push origin v1.2.3
```

Dispatch `publish.yml` from `v1.2.3` in the GitHub Actions UI, with `version`
set to `1.2.3` and `dist-tag` set to `latest`. Alternatively, install and
authenticate the [GitHub CLI](https://cli.github.com/) and run the repository
helper:

```sh
./script/release 1.2.3 latest
```

`script/release` runs the equivalent `gh workflow run publish.yml --ref
v1.2.3` command. To publish a prerelease under another dist-tag with the
helper:

```sh
./script/release 1.2.3-rc.1 next
```

Do not move or replace a release tag. The workflow verifies that the dispatch
ref, checked-out commit, and GitHub SHA all resolve to the same tagged commit.

## Monitor And Verify

The Actions UI shows the dispatched run through validation, build, and
publication. With the GitHub CLI installed and authenticated, the equivalent
monitoring commands are:

```sh
gh run list --workflow publish.yml
gh run watch <run-id>
gh run view <run-id> --log-failed
```

After the run succeeds, query npm rather than relying only on the Actions
status:

```sh
npm view oc2-ai@1.2.3 version dist-tags
npm install --global oc2-ai@1.2.3
oc2 --version
```

Confirm that all native packages at the same version are visible before
announcing the release. If publication stops after only some native packages
are published, fix the external or workflow failure and rerun the same tag,
version, and dist-tag. The publisher verifies matching existing packages and
continues the partial release; do not create a different tag for the same npm
version.
