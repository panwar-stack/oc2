The OC2 CLI by default starts the [TUI](/docs/tui) when run without any arguments.

```bash
oc2
```

But it also accepts commands as documented on this page. This allows you to interact with OC2 programmatically.

```bash
oc2 run "Explain how closures work in JavaScript"
```

---

### tui

Start the OC2 terminal user interface.

```bash
oc2 [project]
```

#### Flags

| Flag                                        | Short | Description                                                             |
| ------------------------------------------- | ----- | ----------------------------------------------------------------------- |
| <nobr><code>{"--continue"}</code></nobr>    | `-c`  | Continue the last session                                               |
| <nobr><code>{"--session"}</code></nobr>     | `-s`  | Session ID to continue                                                  |
| <nobr><code>{"--fork"}</code></nobr>        |       | Fork the session when continuing (use with `--continue` or `--session`) |
| <nobr><code>{"--prompt"}</code></nobr>      |       | Prompt to use                                                           |
| <nobr><code>{"--model"}</code></nobr>       | `-m`  | Model to use in the form of provider/model                              |
| <nobr><code>{"--agent"}</code></nobr>       |       | Agent to use                                                            |
| <nobr><code>{"--port"}</code></nobr>        |       | Port to listen on                                                       |
| <nobr><code>{"--hostname"}</code></nobr>    |       | Hostname to listen on                                                   |
| <nobr><code>{"--mdns"}</code></nobr>        |       | Enable mDNS discovery                                                   |
| <nobr><code>{"--mdns-domain"}</code></nobr> |       | Custom mDNS domain name                                                 |
| <nobr><code>{"--cors"}</code></nobr>        |       | Additional browser origin(s) to allow CORS                              |

## Commands

The OC2 CLI also has the following commands.

### agent

Manage agents for OC2.

```bash
oc2 agent [command]
```

---

#### create

Create a new agent with custom configuration.

```bash
oc2 agent create
```

This command will guide you through creating a new agent with a custom system prompt and permission configuration. Anything you don't allow is denied in the generated agent's frontmatter.

#### Flags

| Flag                                        | Short | Description                                                                                                                                                                                                                |
| ------------------------------------------- | ----- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| <nobr><code>{"--path"}</code></nobr>        |       | Directory to write the agent file to (defaults to global or `.oc2/agent` based on the prompt)                                                                                                                         |
| <nobr><code>{"--description"}</code></nobr> |       | What the agent should do                                                                                                                                                                                                   |
| <nobr><code>{"--mode"}</code></nobr>        |       | Agent mode: `all`, `primary`, or `subagent`                                                                                                                                                                                |
| <nobr><code>{"--permissions"}</code></nobr> |       | Comma-separated list of permissions to allow (default: all). Available: `bash`, `read`, `edit`, `glob`, `grep`, `webfetch`, `task`, `todowrite`, `websearch`, `lsp`, `skill`. Anything omitted is denied. Alias: `--tools` |
| <nobr><code>{"--model"}</code></nobr>       | `-m`  | Model to use, in `provider/model` format                                                                                                                                                                                   |

Passing all of `--path`, `--description`, `--mode`, and `--permissions` runs the command non-interactively.

---

#### list

List all available agents.

```bash
oc2 agent list
```

### attach

Attach a terminal to an already running OC2 backend server started via `serve` or `web` commands.

```bash
oc2 attach [url]
```

This allows using the TUI with a remote OC2 backend. For example:

```bash
# Start the backend server for web/mobile access
oc2 web --port 4096 --hostname 0.0.0.0

# In another terminal, attach the TUI to the running backend
oc2 attach http://10.20.30.40:4096
```

#### Flags

| Flag                                     | Short | Description                                                                |
| ---------------------------------------- | ----- | -------------------------------------------------------------------------- |
| <nobr><code>{"--dir"}</code></nobr>      |       | Working directory to start TUI in                                          |
| <nobr><code>{"--continue"}</code></nobr> | `-c`  | Continue the last session                                                  |
| <nobr><code>{"--session"}</code></nobr>  | `-s`  | Session ID to continue                                                     |
| <nobr><code>{"--fork"}</code></nobr>     |       | Fork the session when continuing (use with `--continue` or `--session`)    |
| <nobr><code>{"--password"}</code></nobr> | `-p`  | Basic auth password (defaults to `OC2_SERVER_PASSWORD`)               |
| <nobr><code>{"--username"}</code></nobr> | `-u`  | Basic auth username (defaults to `OC2_SERVER_USERNAME` or `oc2`) |

### auth

Command to manage credentials and login for providers.

```bash
oc2 auth [command]
```

---

#### login

OC2 is powered by the provider list at [Models.dev](https://models.dev), so you can use `oc2 auth login` to configure API keys for any provider you'd like to use. This is stored in `~/.local/share/oc2/auth.json`.

```bash
oc2 auth login
```

When OC2 starts up it loads the providers from the credentials file. And if there are any keys defined in your environments or a `.env` file in your project.

##### Flags

| Flag                                     | Short | Description                                          |
| ---------------------------------------- | ----- | ---------------------------------------------------- |
| <nobr><code>{"--provider"}</code></nobr> | `-p`  | Provider ID or name to log in to                     |
| <nobr><code>{"--method"}</code></nobr>   | `-m`  | Login method label to use, skipping method selection |

---

#### list

Lists all the authenticated providers as stored in the credentials file.

```bash
oc2 auth list
```

Or the short version.

```bash
oc2 auth ls
```

---

#### logout

Logs you out of a provider by clearing it from the credentials file.

```bash
oc2 auth logout
```

### memory

Index and search repository memory.

```bash
oc2 memory [command]
```

Repository memory stores historical commit data, file activity, and cached summaries for the current repository. Use it to localize likely files for issue-like tasks, regressions, stack traces, migrations, and similar debugging work.

:::caution
Memory is historical. Treat retrieved commits and summaries as localization hints only, and verify against the current working tree before editing.
:::

#### index

Index local git history and generate file summaries.

```bash
oc2 memory index
```

While indexing, oc2 shows foreground progress for repository resolution, commit crawling, index writes, file activity, and optional file summary generation. The final summary output stays unchanged for scripts.

##### Flags

| Flag                                        | Description                                                                                                   |
| ------------------------------------------- | ------------------------------------------------------------------------------------------------------------- |
| <nobr><code>{"--max-commits"}</code></nobr> | Maximum commits to index (default: `7000`)                                                                    |
| <nobr><code>{"--since"}</code></nobr>       | Only index commits since this git date                                                                        |
| <nobr><code>{"--base-commit"}</code></nobr> | Exclusive cutoff commit                                                                                       |
| <nobr><code>{"--cutoff-time"}</code></nobr> | Exclusive ISO cutoff timestamp                                                                                |
| <nobr><code>{"--branch"}</code></nobr>      | Branch or revision to index                                                                                   |
| <nobr><code>{"--github"}</code></nobr>      | Allow linked issue-number lookup from commit messages; use `--no-github` to disable. Indexing is offline-safe |
| <nobr><code>{"--summaries"}</code></nobr>   | Number of top active files to summarize; use `0` to skip summaries                                            |

#### status

Show counts for the active repository memory index.

```bash
oc2 memory status
```

#### search

Search indexed commits or cached file summaries.

```bash
oc2 memory search commit "auth regression" --limit 20
oc2 memory search summary "serialization error" --limit 5
```

#### examine

Inspect a stored commit memory record and its historical diff.

```bash
oc2 memory examine commit <hash> --max-diff-bytes 50000
```

#### view

Show a cached file summary for a repository-relative path.

```bash
oc2 memory view summary packages/oc2/src/example.ts
```

#### clear

Clear repository memory for the current repository, or for a specific repository identity.

```bash
oc2 memory clear
oc2 memory clear --repository github.com/owner/repo
```

#### eval

Evaluate repository memory localization against historical issues.

```bash
oc2 memory eval --issues issues.json
```

##### Flags

| Flag                                        | Description                                                        |
| ------------------------------------------- | ------------------------------------------------------------------ |
| <nobr><code>{"--issues"}</code></nobr>      | Path to historical issues JSON                                     |
| <nobr><code>{"--max-commits"}</code></nobr> | Maximum commits to index per issue (default: `7000`)               |
| <nobr><code>{"--summaries"}</code></nobr>   | Number of top active files to summarize; use `0` to skip summaries |

### models

List all available models from configured providers.

```bash
oc2 models [provider]
```

This command displays all models available across your configured providers in the format `provider/model`.

This is useful for figuring out the exact model name to use in [your config](/docs/config/).

You can optionally pass a provider ID to filter models by that provider.

```bash
oc2 models anthropic
```

#### Flags

| Flag                                    | Description                                                  |
| --------------------------------------- | ------------------------------------------------------------ |
| <nobr><code>{"--refresh"}</code></nobr> | Refresh the models cache from models.dev                     |
| <nobr><code>{"--verbose"}</code></nobr> | Use more verbose model output (includes metadata like costs) |

Use the `--refresh` flag to update the cached model list. This is useful when new models have been added to a provider and you want to see them in OC2.

```bash
oc2 models --refresh
```

### run

Run oc2 in non-interactive mode by passing a prompt directly.

```bash
oc2 run [message..]
```

This is useful for scripting, automation, or when you want a quick answer without launching the full TUI. For example.

```bash "oc2 run"
oc2 run Explain the use of context in Go
```

You can also attach to a running `oc2 serve` instance to avoid MCP server cold boot times on every run:

```bash
# Start a headless server in one terminal
oc2 serve

# In another terminal, run commands that attach to it
oc2 run --attach http://localhost:4096 "Explain async/await in JavaScript"
```

#### Flags

| Flag                                                         | Short | Description                                                                |
| ------------------------------------------------------------ | ----- | -------------------------------------------------------------------------- |
| <nobr><code>{"--command"}</code></nobr>                      |       | The command to run, use message for args                                   |
| <nobr><code>{"--continue"}</code></nobr>                     | `-c`  | Continue the last session                                                  |
| <nobr><code>{"--session"}</code></nobr>                      | `-s`  | Session ID to continue                                                     |
| <nobr><code>{"--fork"}</code></nobr>                         |       | Fork the session when continuing (use with `--continue` or `--session`)    |
| <nobr><code>{"--share"}</code></nobr>                        |       | Share the session                                                          |
| <nobr><code>{"--model"}</code></nobr>                        | `-m`  | Model to use in the form of provider/model                                 |
| <nobr><code>{"--agent"}</code></nobr>                        |       | Agent to use                                                               |
| <nobr><code>{"--file"}</code></nobr>                         | `-f`  | File(s) to attach to message                                               |
| <nobr><code>{"--format"}</code></nobr>                       |       | Format: default (formatted) or json (raw JSON events)                      |
| <nobr><code>{"--title"}</code></nobr>                        |       | Title for the session (uses truncated prompt if no value provided)         |
| <nobr><code>{"--attach"}</code></nobr>                       |       | Attach to a running oc2 server (e.g., http://localhost:4096)          |
| <nobr><code>{"--password"}</code></nobr>                     | `-p`  | Basic auth password (defaults to `OC2_SERVER_PASSWORD`)               |
| <nobr><code>{"--username"}</code></nobr>                     | `-u`  | Basic auth username (defaults to `OC2_SERVER_USERNAME` or `oc2`) |
| <nobr><code>{"--dir"}</code></nobr>                          |       | Directory to run in, or path on the remote server when attaching           |
| <nobr><code>{"--port"}</code></nobr>                         |       | Port for the local server (defaults to random port)                        |
| <nobr><code>{"--variant"}</code></nobr>                      |       | Model variant (provider-specific reasoning effort)                         |
