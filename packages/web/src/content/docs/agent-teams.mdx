---
title: Agent Teams
description: Coordinate multiple OpenCode agents from one lead session.
---

Agent teams let one lead session coordinate multiple background teammate sessions. Each teammate can use its own agent type, prompt, dependencies, mailbox messages, and optional plan approval.

:::caution[Experimental]
Agent teams are experimental. The config, tools, and workflow may change.
:::

---

## Enable

Turn on agent teams in your OpenCode config.

```json title="opencode.json"
{
  "$schema": "https://opencode.ai/config.json",
  "experimental": {
    "agent_teams": true
  }
}
```

After enabling this option, the team tools become available to the lead agent and teammate agents.

---

## Quickstart

Ask OpenCode to create a team when a task has independent pieces of work.

```txt frame="none"
Create an agent team for this refactor. Have one teammate inspect the auth flow,
one teammate inspect the API routes, and one teammate wait for those findings
before implementing the changes.
```

The lead session should create the team, spawn teammates, collect results, and report the final outcome back to you.

You can also be more explicit:

```txt frame="none"
Use agent teams. Create a team named auth-refactor. Spawn an explorer teammate
for auth, another explorer teammate for routes, then spawn an implementer that
depends on both explorers.
```

For non-trivial work, the recommended workflow is:

- create shared tasks for multi-step work
- assign owners before implementation starts
- use dependencies when one task or teammate needs another result
- use plan mode for risky or broad edits
- broadcast scope changes or key discoveries
- run a final `team_report`

## Concepts

### Lead session

The current session becomes the lead when a team is created. Only one active team can exist for a lead session at a time.

The lead coordinates teammates, receives completion updates, approves plans, and can shut down the team.

### Teammates

Teammates are background child sessions. Each teammate has:

- a name
- an agent type
- a role prompt
- optional dependencies
- optional plan approval mode
- a lifecycle: finite `task` by default, or long-lived `daemon`

Teammates send kickoff, progress, blocker, handoff, and completion updates through the team mailbox.

Teammate names must be unique within a team. If older team data contains duplicate names, name-based targeting rejects the ambiguous name and asks for a session ID instead. Session IDs are always authoritative for message recipients, dependency references, and plan decisions.

### Daemon teammates

Daemon teammates are long-lived team members for monitoring, sentinels, rolling checklists, and similar assignments that remain valid until the team shuts down.

Spawn one with `team_spawn` and `lifecycle: "daemon"`. The daemon receives the lead-provided `role_prompt`, performs one initialization work cycle, and then stays registered as `idle` or `running` instead of completing like a finite task teammate.

Use daemon teammates when the assignment is genuinely open-ended, for example:

- watch team coordination and report unresolved blockers
- maintain a rolling integration checklist during a refactor
- inspect incoming mailbox messages for actionable risks
- monitor a running workflow when another tool or user message wakes it

Do not use daemon teammates for ordinary finite research, implementation, review, or verification tasks. Those should remain normal task teammates.

Daemon teammates communicate through the existing mailbox. They should use `team_send_message` when they discover something actionable and `team_get_messages` only at natural boundaries, not in a polling loop. If a daemon needs a periodic timer, file watcher, log tail, or other external trigger, it should report that missing trigger instead of inventing an unbounded loop.

Daemon teammates do not satisfy `depends_on` just because they initialized. Depend on a finite setup or handoff teammate when another teammate needs a completed result.

### Model variants

Teammates inherit the lead session's provider/model by default. When the current model exposes teammate variants, the lead prompt shows the exact variant keys and the lead may choose one per `team_spawn` call.

Use lower reasoning-effort variants only for simple, well-scoped work with precise instructions. Omit `variant` for default behavior, or when the available variant names are uncertain. A teammate that waits on dependencies keeps the variant chosen when it was spawned.

Agent teams do not automatically rank models, optimize cost, or choose variants for you.

### Dependencies

Use `depends_on` or `wait_for` when a teammate must wait for another teammate to complete.

Dependencies can refer to teammate names or teammate session IDs. A blocked teammate starts automatically after all dependencies complete. If a teammate name is ambiguous, use the dependency teammate's session ID.

Daemon teammates are not completion dependencies. A daemon can wait on completed task teammates before initialization, but downstream task teammates are not unblocked by daemon initialization or idle state.

### Mailbox

Team members communicate through mailbox messages. Messages can target:

- `lead`
- a teammate name
- a teammate session ID
- a comma-separated list of recipients

Messages wake idle recipients when possible and are delivered on their next loop iteration.

Mailbox delivery is tracked per recipient. Reports count persisted recipient rows as `pending` or `delivered`; `read` counts appear only if stored delivery state actually records a read transition.

### Shared tasks

The shared task list tracks work across the team. Tasks can have statuses, assignees, and dependency task IDs.

Creating a task records tracking state only. It does not spawn a teammate or wake an existing teammate.

Task tools display short IDs, but `team_task_claim` and `team_task_update` accept either a full task ID or an unambiguous prefix within the current team. Ambiguous prefixes fail without changing task state and list the matching task prefixes.

A pending task can be claimed only after all dependency tasks in the same team are `completed`. Cancelled dependencies do not unblock claims.

### Plan approval

When a teammate starts in plan mode, mutating tools are denied until the teammate submits a plan and the lead approves it.

Plan mode blocks:

- `bash`
- `write`
- `edit`
- `apply_patch`

Rejecting a plan sends feedback back to the teammate and keeps the teammate in plan mode.

Approving a plan is only valid for a teammate that is currently in plan mode. Approval removes the plan-mode deny overlay, switches the teammate to implementation mode, sends an approval message, and wakes the teammate. Rejection keeps the plan-mode restrictions intact.

## Tools

These tools are used by OpenCode agents when agent teams are enabled.

| Tool                | Purpose                                                                                 |
| ------------------- | --------------------------------------------------------------------------------------- |
| `team_create`       | Create a team for the current lead session.                                             |
| `team_spawn`        | Spawn a background teammate session.                                                    |
| `team_send_message` | Send a mailbox message to one or more recipients.                                       |
| `team_broadcast`    | Broadcast a message to all active teammates, and to the lead when called by a teammate. |
| `team_get_messages` | Read pending mailbox messages for the current team session.                             |
| `team_task_create`  | Create a shared task list item.                                                         |
| `team_task_list`    | List shared team tasks.                                                                 |
| `team_task_claim`   | Claim a pending task whose dependencies are complete.                                   |
| `team_task_update`  | Update a task status or assignee.                                                       |
| `team_plan_submit`  | Submit a teammate plan for lead approval.                                               |
| `team_plan_decide`  | Approve or reject a teammate plan.                                                      |
| `team_report`       | Generate a post-session effectiveness report with usage metrics and findings.           |
| `team_shutdown`     | Close the active team and cancel active teammates.                                      |

`team_report` uses persisted team state for deterministic findings, including blocked teammates, pending mailbox rows on closed teams, shallow usage, missing task lists, missing final reports for non-trivial completed teams, and daemon lifecycle issues such as daemon errors or daemons left active after shutdown.

## TUI

Agent teams add a team sidebar section and a team panel.

| Action              | Keybind      | Description                                                |
| ------------------- | ------------ | ---------------------------------------------------------- |
| `team_panel_toggle` | `<leader>v`  | Open the team panel.                                       |
| `team_task_list`    | `<leader>k`  | Open the team panel on the task list.                      |
| `team_cycle_lead`   | `<leader>up` | Navigate from a teammate session back to the lead session. |

The team panel shows the team goal, teammate statuses, daemon state, pending permissions or questions, shared tasks, and team messages. It can also shut down the team.

Regular child-session navigation still works for teammate sessions:

- `session_child_first`: go to the first teammate session
- `session_child_cycle`: go to the next teammate session
- `session_child_cycle_reverse`: go to the previous teammate session
- `session_parent`: return to the lead session

## HTTP API

The OpenCode server exposes read and shutdown endpoints for agent teams.

| Method | Path                        | Description                                     |
| ------ | --------------------------- | ----------------------------------------------- |
| `GET`  | `/team?sessionID=<session>` | Get the active team for a lead session.         |
| `GET`  | `/team/{teamID}`            | Get a team by ID.                               |
| `GET`  | `/team/{teamID}/eval`       | Get a team evaluation report.                   |
| `GET`  | `/team/{teamID}/tasks`      | List shared team tasks.                         |
| `GET`  | `/team/{teamID}/messages`   | List team messages.                             |
| `POST` | `/team/{teamID}/shutdown`   | Shut down the team and cancel active teammates. |

See [Server](/docs/server#team) for the full server API list.

## Troubleshooting

### Agent teams are not enabled

Add `"experimental": { "agent_teams": true }` to `opencode.json`, then restart OpenCode so the config is reloaded.

### No active team

Create a team before spawning teammates or sending team messages. The current session must be the lead session, or an existing teammate session.

### Unknown agent type

The teammate's `agent_type` must be a configured agent. See [Agents](/docs/agents) for built-in and custom agents.

### Dependency teammate not found

Dependencies must match an existing teammate name or teammate session ID. Spawn the dependency teammate first.

### Teammate is waiting

A teammate with dependencies remains blocked until all dependency task teammates complete. Daemon initialization does not unblock dependents. A teammate in plan mode waits until the lead approves or rejects the submitted plan.

### Team no longer needed

Use `team_shutdown` or the team panel shutdown action. This closes the team and cancels active teammates, but it does not cancel the lead session.
